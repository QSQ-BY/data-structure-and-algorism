## 昨天晚上：用链表实现队列

链表队列这一版没有容量限制，所以它和前一天写的循环队列很不一样。循环队列的重点在 `head`、`tail` 和取模，链表队列的重点则落在 `head`、`tail` 两个指针的维护上。

底层链表用了无头链表：

```cpp
typedef struct linklist{
    node* head;
    node* tail;
    int count;
}linklist;
```

入队其实就是链表尾插。空链表时，`head` 和 `tail` 都要指向新节点；非空时，让旧尾节点的 `next` 指向新节点，再更新 `tail`。这个地方如果只改一个指针，队列就会断掉。

出队就是删头节点。这里最需要小心的是最后一个节点被删掉以后，`tail` 也要置空。否则队列明明已经空了，尾指针还指着一块被释放过的内存，后面再操作就很危险。

我在队列外层又保留了一个 `queue.count`，底层链表也有 `linklist.count`。这有一点重复，但测试时反而能看出两个层级的状态是否同步。`push` 成功后两个 count 都加一，`pop` 成功后两个 count 都减一。表格里同时打印 `QCount` 和 `LCount`，能很快发现有没有漏维护。

这一版让我重新感觉到，队列不是数组或者链表本身。数组可以实现队列，链表也可以实现队列。真正的队列语义是先进先出：从一端进，从另一端出。底层结构只是支撑这种语义的材料。

## 今天上午：数组栈和状态同步

今天先写了一个数组栈。结构很直接：

```cpp
typedef struct stack{
    int* data;
    int top;
    int size;
    int count;
}stack;
```

数组栈这一版最值得注意的是状态维护。`top` 指向当前栈顶，`count` 记录元素个数，`size` 限制最大容量。压栈时先移动栈顶再写入数据，弹栈时只需要让栈顶回退，并同步减少元素数量。

后面我给它补了测试。测试里跑了建栈、空栈判断、连续压栈、满栈压入失败、弹栈、弹到空栈后再弹。最后结果是 `Passed: 10 / 10`。

这个过程让我觉得，栈的代码不难，难的是每个状态都要能对上。`top` 指向当前栈顶，`count` 记录元素个数，压栈时两个都变，弹栈时两个也都变。只要有一个地方没同步，结构就会慢慢歪掉。

## 练习：括号匹配，把栈从代码里抽出来

括号匹配这一题很有意思。题面看起来是在判断字符串是否合法，真正写起来却是在模拟一段栈的历史。

左括号入栈，右括号拿来和栈顶比较。如果类型一致，就把栈顶弹出；如果不一致，说明这个右括号解决不了当前最里层的问题。最后栈为空，说明所有左括号都找到了对应的右括号。

一开始我也踩了几个坑。比如 `match` 里面把字符和字符串字面量混在一起写，`')'` 写成了 `")"`；后来又把输入字符串 `s[top]` 当成了栈顶，其实真正的栈是 `stack[top]`。还有一个边界是，遇到右括号时必须先判断栈是不是空的，不能先访问 `stack[top]` 再检查。

最终测试覆盖了这些用例：

```text
()        true
()[]{}    true
(]        false
([])      true
([)]      false
)         false
(((       false
({[]})    true
({[)]}    false
```

运行结果是 `Passed: 9 / 9`。看到这张表的时候，我心里其实比只看到力扣通过更踏实。因为每个错误情况都摆出来了，尤其是右括号开头、左括号残留、交叉匹配这几类，正好对应栈最容易漏掉的边界。

## 关于栈的一点想法

今天的括号匹配问题，和之前做快乐数时的感觉有点像。快乐数让我意识到，链表不一定真的长成链表；只要一个状态能推出下一个状态，就可能藏着链表和环的结构。括号匹配则让我对栈有了类似的感觉。

栈的入栈和出栈，可以被抽象成左括号和右括号。左括号表示一个问题、一个事件、一个上下文被打开；右括号表示它被解决、被关闭、被收束。一对括号包起来，就是一个完整事件。

再往外看，括号序列本质上是一种从整体到部分的主从结构。一个大问题里包含几个小问题，一个主函数里调用不同的子函数，一个任务推进过程中又不断打开新的子任务。解决这类有包含关系的问题时，栈就很自然。最新出现的问题要最先被解决，最里面的括号要最先闭合。

这个想法让我挺兴奋的。因为它说明数据结构不是一套死记硬背的模板，而是一种描述世界的方式。链表可以描述状态如何向下一个状态流动，队列可以描述先来后到的处理顺序，栈可以描述嵌套、包含和回到上一层。
