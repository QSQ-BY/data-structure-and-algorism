# 2026-06-07：开始写堆和优先队列

今天开始进入“堆和优先队列”这一章。昨天合并果子的时候刚碰到“每次取最小的两个元素”这种需求，当时只是知道优先队列会更适合。今天自己动手写基础结构以后，这个数据结构终于不再只是一个 STL 名字，而是能落到数组下标和调整过程里了。

我这次写的是大顶堆。结构体里保留了原始数组指针、偏移后的访问指针、容量和当前元素个数。堆顶放在 `data[1]`，这样父节点和左右孩子的关系可以直接写成：

```cpp
#define FATHER(i) ((i)/(2))
#define LEFT(i) ((i)*(2))
#define RIGHT(i) ((i)*(2)+(1))
```

这个下标设计比从 0 开始更适合手写堆。看公式的时候很清楚：当前节点向上找父亲，或者向下找两个孩子，都不需要额外处理太多偏移。今天真正要记住的不是宏本身，而是堆的形状藏在数组下标里。它看起来像一个普通数组，但每一个位置都在一棵完全二叉树里。

## 上滤和下滤是两种相反的修复

`push` 的过程比较顺：新元素先放到数组最后，再一路和父节点比较。如果新元素比父节点大，就交换，直到它不再破坏大顶堆的性质。这个过程叫上滤，我能把它理解成“新来的节点从叶子往上找自己的位置”。

`pop` 的方向正好反过来。删除堆顶后，先把最后一个元素挪到根节点，再从根往下调整。每一轮都在当前节点、左孩子、右孩子中找最大的那个。如果最大的不是当前节点，就交换，并继续往下走。这里比上滤更容易漏条件，因为右孩子可能不存在，循环边界也要盯紧。

写完以后，我对堆的“不变量”有了更具体的感觉：不是每次操作后整个数组都有序，而是每个父节点都要压住自己的孩子。这个区别很重要。如果期待数组从大到小排列，就会误解堆；真正应该看的，是堆顶是不是当前最大值，以及每一次调整后父子关系有没有被修回来。

## 从合并果子接到优先队列

昨天的合并果子还停留在“反复拿最小两堆”的题目层面，今天再看优先队列，就能把这个需求和结构对应起来。堆不追求所有元素整体有序，但它保证最值一直在堆顶。这样每次取最值时，不需要重新排序整个数组，只要在局部修复堆的性质。

这也让我重新理解了“优先队列”这个名字。它不是普通队列那种先来先服务，而是让优先级最高的元素先出来。大顶堆里，优先级可以理解成数值更大；如果以后改成小顶堆，比较规则反过来，就能服务合并果子这类每次取最小值的问题。

今天没有把所有细节都铺开重构。对我来说，当前更重要的是先把优先队列的基本操作跑起来，再把它和合并果子、哈夫曼编码重新接上：同样是反复取最值，用手写堆来做，应该比每次排序更接近这个结构本来的用途。
