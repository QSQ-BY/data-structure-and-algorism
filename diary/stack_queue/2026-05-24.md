# 2026-05-24：栈序列练习，终于把“模拟”这件事想清楚了一点

我今天基本都在和栈打交道。做的题不算分散：退格字符串、火车出站序列、LeetCode 946 验证栈序列。它们表面上题面不同，一个是退格字符串比较，一个是火车站，一个是 push/pop 序列，但我写到后面才发现，核心其实都落在同一个问题上：当前栈里到底保存了什么，栈顶现在能不能被弹出。

## 退格字符串：先把“栈的长度”想明白

一开始看 `lc844_string_with_delete.cpp`，我还是沿着数组下标的习惯去写，用了一个 `top` 去表示栈顶。这里最容易混的是：`top++` 只是把变量变成了 0，并不会让 `string` 真的多出一个第 0 位。

所以我后来意识到，`stack[top] = s[i]` 这种写法在空字符串上是不成立的。真正的压栈动作应该是：

```cpp
stack.push_back(s[i]);
```

我后来还注意到一个细节：空文本遇到 `#` 时，不是处理结束，而是继续保持空文本。因此 `"#a"` 应该还原成 `"a"`。我把测试补进去之后，这个边界很快就露出来了。这个用例比题面三个样例更尖锐，因为它直接检查“空栈退格”到底是 `break` 还是 `continue`。

这件事让我重新确认了一点：如果用 `vector` 或 `string` 模拟栈，最好先相信容器自己的长度。`push_back()`、`pop_back()`、`back()` 已经把栈顶关系表达得很清楚了，自己再维护一个 `top`，反而多了一个要同步的状态。

## 火车出站：从判断一个排列，到输出前 20 个方案

我今天花时间最多、也最难的题无疑是 `hz263_train.cpp`。火车进站问题写得很久，中间反复改判断函数、补测试、再把递归枚举接上去，确实有点磨人。但也正因为它复杂，我对栈的理解又被压实了一层：不是只会说“先进后出”，而是要能把进站、等待、出站这些动作真的模拟出来。

最开始我想写一个函数，输入一个全排列，判断它是不是合法出栈序列。这个判断题和火车题的关系很直接：火车只能按 `1,2,3...n` 进站，出站序列给定以后，就看能不能通过一个栈模拟出来。

我最后把判断函数稳定成了这种结构：

```cpp
vector<int> stack;
int next_train = 1;

for(int i = 0; i < n; i++){
    while(next_train <= n && (stack.empty() || stack.back() < arr[i])){
        stack.push_back(next_train);
        next_train++;
    }

    if(!stack.empty() && stack.back() == arr[i]){
        stack.pop_back();
    }else{
        return false;
    }
}
return true;
```

我现在觉得这段代码里最关键的是 `next_train`。它表示下一辆还没进站的车。对于目标出站车 `arr[i]`，如果栈顶还没到它，我就继续让车按顺序进站；如果栈顶等于它，就弹出；如果栈顶已经不是它又没法再靠入栈改变局面，那这个排列就不合法。

我后来给这个函数补了完整小规模测试：枚举 `n=1` 到 `n=4` 的所有排列，一共 33 个。这个测试挺有价值，因为它不是只挑几个顺眼的例子，而是把小范围里的合法和非法情况都扫了一遍。之前 `{1,2}` 这种最基础的合法序列就曾经暴露过空栈访问的问题。

在判断函数稳定之后，我又把题目补完整：读入 `n`，递归生成排列，筛出合法出站序列，输出前 20 种方案。这里也顺便复习了一遍用递归函数实现排列型枚举的算法。递归枚举本身我之前写过，但这次要把它和合法出栈判断接起来，就不能只停在“能枚举所有排列”上。`arr`、`flag`、`count` 如果每次递归都重新创建，前面填过的位置就丢了。改成类成员之后，递归才真正有了连续的状态。

这题花的时间极长，但收获也确实满。它把栈模拟、全排列枚举、边界测试和题目输出要求都串在了一起，比单独做一道“判断是否合法”的小题更完整。

最后验证了题面样例：

```text
n = 3
123
132
213
231
321
```

我也验证了 `n=4` 的输出，结果和样例一致。跑 `n=20` 时也能很快给出前 20 种，因为我在递归里加了 `count >= 20` 的停止条件。

## LeetCode 946：同样是栈序列，但入口不一样

我之后又看了 `lc946_judge_stack.cpp`。这题和火车出站很像，但有一个差别：火车题默认入栈顺序是 `1..n`，而 LC946 直接给了 `pushed` 数组。

我现在觉得最清楚的写法是：

```cpp
for(int i = 0; i < pushed.size(); i++){
    stack.push_back(pushed[i]);

    while(!stack.empty() &&
          popped_count < popped.size() &&
          stack.back() == popped[popped_count]){
        stack.pop_back();
        popped_count++;
    }
}
```

我把它理解成：每压入一个元素，就尽量把当前能弹出的元素全部弹掉。这个模型比火车题更直接，我不需要自己推下一个编号，只需要按 `pushed` 顺序走。

这里还有一个小插曲。因为我刚写完复杂的火车进站问题，脑子里还停留在“要不断判断什么时候继续入栈、什么时候出栈”的状态，所以一开始写 LC946 的时候，我把它想得太复杂了。我的入栈操作也是在 `for` 循环里面再套一层 `while`，加了很多条件判断。虽然这样写在力扣那通过了，但我再仔细看，发现那个 `while` 实际上每轮最多执行一次：压入 `pushed[i]` 之后，栈顶就等于 `pushed[i]`，循环条件马上不成立。它虽然写得绕，但本质仍然是在按 `pushed` 顺序入栈，再尽可能匹配 `popped`。

看完标准答案以后，这个感觉更明显：我在`while`里加上那一堆约束条件之后，其实就等价于一个简单的 `push_back` 入栈操作。每次入栈后，判断一下能不能出栈就够了。不是每道栈序列题都需要像火车进站那样先自己推“下一辆车是谁”，LC946 已经把入栈顺序放在 `pushed` 里了，我只要照着它走。

这个地方挺值得记下来。判断一段代码不能只看形状像不像标准答案，还要看循环条件到底允许它运行几次。今天这一下算是给我也提了个醒。

## 今天真正练到的东西

我今天不是简单多写了几道栈题，而是反复碰到同一组概念：

```text
栈里保存的是还没出场的元素。
栈顶代表当前唯一能被弹出的元素。
目标序列用一个指针向前推进。
每次入栈后，都要尽可能检查能不能弹栈。
```

我把这几句话放到退格字符串里，看的是字符能不能被删除；放到火车题里，看的是当前车能不能出站；放到 LC946 里，看的是 `popped[popped_count]` 能不能被匹配。

我今天最大的收获大概是：栈题里的“模拟”不是照着题面机械翻译动作，而是要先确定不变量。只要能说清楚栈里保存的是什么，栈顶表示什么，哪个指针代表下一个目标，代码就会稳很多。反过来，如果这些东西没想清楚，就会下意识加 `top`、加计数器、加循环，最后状态越来越多，自己也说不清谁该负责什么。

接下来再写这类题，我想尽量先用 `back()` 和 `empty()` 把模型写干净。只有真的需要数组下标时，再维护额外的 `top`。今天这些题已经够说明问题了：少一个手动同步的状态，代码就少一个出错入口。
