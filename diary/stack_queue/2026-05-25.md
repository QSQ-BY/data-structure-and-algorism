# 2026-05-25：两道栈队列题，真正难的是状态别乱

今天做的题不算多，就两道：一道是最长美观括号序列，另一道是设计循环队列。数量上看起来很轻，但写下来并不算水过去。尤其是第一题，题面只是在普通括号匹配后面多加了一句“找最长的连续子序列”，难度一下就换了一层。

## 最长美观括号序列：匹配不难，连续最长才难

这题一开始看起来像括号匹配的扩展版。普通括号匹配只要判断一整个序列是否合法，栈里保存没匹配的左括号，遇到右括号就看栈顶能不能对上。这里多了三种括号：`()、[]、{}`，但这部分还算直接，栈顶和当前右括号类型一致就配对，不一致就说明这里断开。

真正让我绕进去的是“最长美观的连续一段”。这句话看起来只是题目要求多了一点，但它改变了问题性质。原来只要回答“合法或不合法”，现在要在一串可能很乱的括号里，把能拼起来的合法片段找出来。

我先用了 `match` 数组记录配对关系：如果第 `i` 位和第 `j` 位匹配，就让 `match[i] = j`，`match[j] = i`。这个设计本身挺清楚，后面可以通过 `match` 看每个位置有没有配对，也能知道它的另一端在哪里。问题出在怎么根据这些配对信息求最长长度。

我一开始想着看到匹配位置就跳转，但这里很容易把指针跳回去，导致扫描方向乱掉。后来我才把这个点想清楚：如果从左括号端点开始统计，就应该跳到 `match[i] + 1`，也就是整段匹配括号的右端点后一位。扫描指针应该一直往右走，不能从右端点再跳回左端点。

今天补的测试也挺有用。像 `([])` 这种嵌套结构能测出单个合法块，`[[[[]]{}]]` 能测出题目样例和复杂嵌套，但 `()[]{}`、`([{}])[]{}`、`{}[()](())` 这种才是真的在检查规则 3：如果 `A` 和 `B` 都美观，那么 `AB` 也美观。之前如果只盯着一对括号或者一个嵌套块，就会漏掉这种连续拼接。

这也是今天最明显的感受：括号匹配的“栈”只是第一层，最长连续合法段才是第二层。多出来的不是一行输出，而是一个新的状态统计问题。

## 循环队列：小菜一碟，也不能少维护状态

第二题是实现循环队列。对现在的我来说，这题确实没那么吓人了。队列的 FIFO、`Front()`、`Rear()`、`enQueue()`、`deQueue()` 这些接口都很直观。我这次选的是预先建好 `k` 个结点，把它们连成一个环，用 `head` 指向队首，`tail` 指向队尾，入队时写到 `tail->next`，再把 `tail` 往后挪。

这个模型我能想出来，也能说清楚，所以写主体时比较顺。但最后还是忘了维护 `count`。这不是算法不会，而是状态量没同步：入队成功以后要 `count++`，出队成功以后要 `count--`。少了这两句，`isEmpty()` 和 `isFull()` 的判断就全都失真，`Front()`、`Rear()` 也会跟着错。

最近确实老犯这种小错误。今天这次我不想只把它记成“粗心”，因为光说粗心没什么用。更准确地说，是我在写数据结构时还需要把不变量放在眼前：

```text
count 表示当前队列里的元素个数。
0 <= count <= size。
enQueue 成功时 count 增加。
deQueue 成功时 count 减少。
isEmpty 和 isFull 都只相信 count。
```

只要这个不变量明确，代码就没有什么玄学。忘记 `count` 的本质，是我写完指针移动以后，没有回头检查“状态量是否和结构变化同步”。

## 今天留下的感觉

今天这两题放在一起还挺有意思。最长括号序列让我看到：题目条件只要多一句，原来的简单匹配就可能变成区间统计。循环队列则提醒我：题目本身不难的时候，反而更容易在基本状态维护上松一下。

我今天没有刷很多题，但这两题都指向同一件事：写数据结构时，不能只盯着操作动作，还要盯着动作之后哪些状态必须保持一致。栈里保存什么、`match` 表示什么、`count` 代表什么，这些变量的含义如果提前说清楚，后面调试会少很多绕路。

明天打算再写一道竞赛难度的题给栈和队列大火收汁了，再后面可能还会刷几道相关的题，但从后天开始学习的中心就放到“树”上面了。
