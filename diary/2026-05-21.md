# 2026-05-21：用顺序表封装循环队列的一天

今天主要围着一个问题打转：怎么用已经封装好的顺序表，再去封装一个队列。

这个问题一开始看起来不大。队列嘛，有 `push`、`pop`、`front`，再加上 `head` 和 `tail`，好像很快就能写完。但真正写到顺序表那一层时，我发现自己把两种思路混在了一起：一边想用顺序表的 `insert`，一边又想写循环队列。

这两件事看着都合理，放在一起却不太对劲。

## 从 `insert` 到 `set`

最开始我给顺序表设计了 `insert` 函数，想法是让队列不要直接碰底层数组，而是通过已经封装好的接口完成入队。这个方向本身没问题。问题在于，`insert` 的语义是“插入”：从某个位置开始，后面的元素要整体右移，顺序表的 `count` 也会增加。

但循环队列里的 `tail` 不是普通顺序表里的插入位置。它更像是“下一次写入的位置”。当 `tail` 走到数组末尾后，它会绕回到 0。这个时候再调用 `insert`，顺序表会按线性结构移动元素，而循环队列本来不应该移动任何已有元素。

这个地方我卡了一下。因为表面上看，`tail` 指向队尾，插入和赋值好像差别不大。但仔细推一遍状态就能发现，`insert` 会改变底层顺序表的结构，`set` 只是改某个下标上的值。循环队列需要的是后者。

所以我把底层接口改成了：

```cpp
int set(vector* v, int pos, int val) {
    if (v == nullptr or pos < 0 or pos >= v->size) return 0;
    v->data[pos] = val;
    return 1;
}
```

这样队列仍然没有直接写 `q->data->data[q->tail] = val;`，封装关系还在，只是接口语义变得更准确了。

## 队列层只负责队列

调整之后，结构清楚了很多。

`vector` 只负责固定数组的读写：

```cpp
set(v, pos, val);
seek(v, pos);
```

`queue` 负责维护循环队列状态：

```cpp
q->tail = (q->tail + 1) % q->size;
q->head = (q->head + 1) % q->size;
```

入队时，先把值写到 `tail` 位置，再移动 `tail`。出队时，不需要真的删除数组里的元素，只要移动 `head`。队列是否为空、是否已满，都交给 `q->count` 来判断。

现在这几个函数的职责比较明确：

```cpp
int push(queue* q,int val){
    if(q==nullptr or q->size == q->count) return 0;
    if(set(q->data,q->tail,val) == 0) return 0;
    q->tail = (q->tail+1)%q->size;
    q->count++;
    return 1;
}

int pop(queue* q){
    if(q==nullptr or q->count==0) return 0;
    q->head = (q->head+1)%q->size;
    q->count--;
    return 1;
}

int front(queue* q){
    if(q==nullptr or q->count==0) return -1;
    return seek(q->data,q->head);
}
```

这里还有一些工程细节以后可以继续补，比如 `malloc` 失败检查、`size <= 0` 的处理，以及 `front` 用 `-1` 表示错误时会和真实数据冲突。不过今天这一步的重点不是把它写成工业级容器，而是把数据结构模型理顺。

## 给队列补测试

后面我又给 `test01()` 补了一段表格测试。这个测试不是只跑一两个 happy path，而是专门覆盖了几个容易出错的状态：

- 空队列调用 `front`
- 空队列调用 `pop`
- 连续入队直到满
- 满队列继续入队失败
- 出队后再入队，让 `tail` 绕回
- 连续出队直到重新变空

测试输出里同时打印 `Expected`、`Actual`、`Front`、`Head`、`Tail`。这样比单纯看返回值直观很多，尤其是循环队列这种结构，指针位置比结果本身更能说明问题。

有一行我挺喜欢：

```text
| 8    | push 40        | 1        | 1      | 20    | 1    | 1    |
```

这说明队列在出掉 `10` 之后，又把 `40` 写回了数组前面的位置。`tail` 绕回了，但 `front` 仍然是 `20`。这就是循环队列应该有的样子。

## 今天真正学到的东西

今天最有价值的地方，不是把某个语法错误修掉了，而是重新想清楚了“封装”到底是什么意思。

封装不是把所有底层访问都藏起来就完事了。更重要的是，每一层提供的接口要和上一层的模型对得上。顺序表的 `insert` 是线性表语义，循环队列需要的是固定位置写入语义。它们都可以被封装，但不能随便替换。

换句话说，`insert` 和 `set` 的区别不是写法问题，而是数据结构模型问题。

今天的代码还不算完美，但比一开始清楚多了。下一次再回头看这份队列实现，我应该会先补创建函数的异常处理，再考虑要不要把 `vector.count` 从这个版本里拿掉。因为在现在的循环队列设计里，真正表示元素数量的是 `q->count`，不是底层顺序表的 `count`。

能把这个边界分清楚，今天就没白写。
