# 2026-05-30：线索化二叉树，自己写一遍才知道哪里没有懂

昨天跟着老师把二叉树遍历和线索化的代码打了一遍，最后还给自己留了一个任务：今天把代码遮起来，完完整整地自己写一次。

今天真写起来，只能说是一场苦行。

昨天听课的时候虽然已经觉得吃力，但老师的代码摆在那里，思路总归有一条现成的路可以跟着走。今天从节点定义、随机插入、三种递归遍历一路写到中序线索化，轮到自己决定每一行应该放在哪里时，很多原本以为已经理解的地方又散开了。最后还是借助 AI 一点点 debug，才把中间卡住的地方重新理顺。

线索化最难的部分，还是理解 `pre_node` 到底代表谁。它不是树结构里固定的某个节点，而是随着中序遍历向前移动，始终保存“刚刚访问过的节点”。递归处理完左子树、回到当前节点时，`pre_node` 才恰好是当前节点的中序前驱。当前节点如果没有左孩子，就可以把空出来的左指针改成前驱线索；`pre_node` 如果没有右孩子，也可以把它的右指针改成指向当前节点的后继线索。想清楚这个过程以后，我才真正明白为什么线索化代码必须夹在左右子树的两次递归之间。

今天我还绕了很久 `tag` 的含义。我一开始会觉得：最左侧节点的前驱本来就是空，最右侧节点的后继也本来就是空，为什么还要专门把它们标记成线索？后来才意识到，`tag` 记录的不是“指针是不是空”，而是“这个指针应该按照什么语义解释”。`tag == 0` 表示真实孩子边，`tag == 1` 表示前驱或后继线索。即使线索最终指向 `nullptr`，它依然是一条线索。这个区别看起来细，但后面用循环寻找最左节点时会直接影响判断条件。

非递归中序遍历也让我重新走了一遍中序遍历的逻辑。完成线索化后，从整棵树最左侧的节点出发，每次寻找当前节点的后继：

```cpp
if(root->right_tag == 1) return root->right_child;
```

如果右指针已经是线索，直接沿着它走就行。如果右指针仍然表示真实右子树，就要进入右子树，再一路找到其中最靠左的节点。原因其实就是最基础的中序规则：左子树、当前节点、右子树。进入右子树以后，最先访问到的一定是那棵子树中最靠左的节点。

虽然但是，到目前为止，我还没有真正感受到线索化有多大的实际用处。它确实把遍历从递归改成了非递归，也不再需要额外的栈空间。但有一说一，以我现在看到的使用场景，线索化消耗的脑细胞和它带来的收益完全不成正比，哈哈。以后真遇到普通的遍历问题，我大概还是会用寥寥的几行递归解决，而不是先完成一整套线索化，再写非递归遍历。

不过这也让我反过来感受到了递归的方便。刚开始学递归时，我总觉得它难以理解：函数为什么能调用自己，返回时又到底回到了哪里。现在再看，递归反而成了节省脑细胞的利器。二叉树本身就有递归结构，用递归去描述它，比手动维护一套前驱、后继和 tag 轻松太多。

昨天我觉得自己大概理解了线索化，今天才发现那种理解还停留在“看别人写的时候能跟上”。真正独立写一遍以后，才会遇到更具体的问题：最左节点应该怎么判断，最后一个节点为什么要处理空后继，`tag` 到底标记的是指针值还是指针含义，真实右子树和后继线索又该怎么区分。AI debug 的过程并不轻松，但每追问一次，原本模糊的一块就会清楚一点。

今天没有急着继续往后赶进度。能把这段线索化代码从头写出来，再把非递归中序遍历为什么成立讲明白，已经够费劲了。接下来我还想找一棵结构固定的小树，拿纸把每个节点的前驱、后继、`left_tag` 和 `right_tag` 全部标出来。随机生成的树适合跑代码，但真要把脑子里的线索理顺，还是得慢一点。我也希望后面能遇到一些真正适合线索化的题目，让我看看它除了把递归改写成循环之外，还能发挥什么作用。
