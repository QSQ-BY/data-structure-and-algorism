说实话，今天的复现比我想象中顺利。大部分代码都能自己写出来，遇到卡壳时也会回头看一下昨天留下的思路和代码。写完以后还是出了 bug，不过和第一次接触这段代码相比，我已经不是照着现成实现往下抄了。自己决定变量怎么定义、每一步放在哪里，感受确实不太一样。

## 二叉树转广义表：递归部分已经顺手一些了

二叉树转广义表仍然沿用昨天的写法。先输出当前节点的数据，再递归处理左右子树。如果当前节点还有孩子，就补上括号和逗号，把左右子树的位置表示清楚。

```cpp
void __serialize(node* root){
    if(root == nullptr) return;
    len += sprintf(buff+len,"%d",root->data);
    if(root->lchild == nullptr and root->rchild == nullptr) return;
    len += sprintf(buff+len,"(");
    if(root->lchild) __serialize(root->lchild);
    len += sprintf(buff+len,",");
    if(root->rchild) __serialize(root->rchild);
    len += sprintf(buff+len,")");
}
```

这一部分今天没有卡太久。树本身就是递归结构，广义表里也保留了嵌套关系，所以只要把“当前节点、左子树、右子树”的顺序理清楚，代码就能比较自然地写下来。

不过，写到 `len += sprintf(buff + len, ...)` 时，我还是卡了一下，又翻回之前的代码看了看。这里的 `buff + len` 指向缓冲区中下一段内容应该开始写入的位置，`sprintf` 的返回值则是本次写入的字符数。把返回值继续累加到 `len` 上，下一次就能接着字符串末尾追加内容。这个技巧看懂时不难，自己重新写的时候却还没有完全形成习惯。

## 广义表转二叉树：真正容易忘的是接入时机

反序列化还是更难一些。扫描字符串时，我需要根据当前字符决定下一步动作：读到数字就创建节点，读到左括号就把刚创建的节点压栈，读到逗号就切换到右子树，读到右括号就弹栈。

今天重新梳理时，我又在一个地方卡住了：新节点到底应该什么时候接到树上？

新节点创建出来以后，如果栈里还有尚未处理完的父节点，就要立刻根据 `flag` 把它接到栈顶节点的左边或右边。栈顶保存的是当前正在构建的父节点，`flag` 保存的是接下来应该填左子树还是右子树。只记得“遇到括号要压栈、遇到右括号要弹栈”还不够，节点接入父节点的时机也必须想清楚。

```cpp
p = get_new_node(number);
if(root == nullptr) root = p;
if(flag == 0){
    if(top != -1) stack[top]->lchild = p;
}else{
    if(top != -1) stack[top]->rchild = p;
}
```

这也让我发现，自己对栈的理解还不算深入。以前写括号匹配时，栈里存的是还没有配对的左括号。今天栈里存的是还没有构建完成的父节点。形式很像，但要真正写出来，还是得明确每次压栈、弹栈以后，栈顶元素在当前问题里代表什么。

## 变量不是一开始就能全部想齐

这段代码里的变量不少。我一开始总想把所有东西一次性定义完整，但真写起来很难做到。现在我的经验是，先把算法思路梳理清楚，再把确定会用到的变量列出来：

- `stack`：保存尚未构建完成的父节点；
- `top`：记录栈顶位置；
- `scode`：给有限状态机分发任务；
- `flag`：标记接下来处理左子树还是右子树。

后面写到具体流程时，再补上 `p`、`root` 和 `number`。`p` 永远指向刚创建的新节点，`root` 保存整棵树的根节点，`number` 用来拼出可能有多位的节点数据。

以前我会觉得，写代码前没有一次性列全变量，说明思路不够完整。今天写完以后，我觉得这件事不用太僵硬。先抓住稳定的结构，再在实现过程中补充局部变量，反而更符合真实的编码过程。前提是每个变量的职责要说得清楚，不能写到后面自己也忘了它为什么存在。

## 开始学习哈夫曼编码

今天还学习了定长编码、变长编码和哈夫曼编码的理论基础。

定长编码比较直接：每个字符都使用相同长度的二进制串表示。变长编码则会根据字符出现频率分配不同长度的编码，常见字符尽量短，出现较少的字符可以长一些。哈夫曼编码就是在这个思路上，根据权值不断合并较小的节点，最终得到一棵哈夫曼树，再从树上的路径生成编码。

目前我还停留在理论部分。明天开始写哈夫曼编码的代码实现时。今天重新复现序列化代码时，我已经感受到“理解大致思路”和“自己决定每一步怎么落到变量上”之间还有一段距离。明天写哈夫曼编码，大概也会遇到同样的问题。
