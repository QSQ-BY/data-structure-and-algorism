# 2026-06-05：层序遍历熟了，递归边界还得慢慢拆

今天代码量不算多，主要写了两道 LeetCode 树题：锯齿形层序遍历和判断树的子结构。题目都没有特别大的算法跨度，但正好把这几天反复练的 BFS 和递归又接了一遍。

锯齿形层序遍历非常简单。普通层序遍历先把每一层存下来，然后把奇数下标的层翻转一下，就能得到题目要求的左右交替顺序。核心还是那套 BFS 写法：队列里先放根节点，每一轮用 `cnt = q.size()` 固定当前层的节点数，再把当前层的节点依次取出来，把它们的左右孩子放进队列。

昨天重写层序遍历时，我还在反复确认 `cnt` 和 `q.front()` 这些细节。今天再写锯齿形遍历，手感明显顺了很多。真正要想的地方只剩下“哪一层需要翻转”：按数组下标看，第 0 层保持从左到右，第 1 层翻转，第 2 层再保持原样。这个点确认以后，代码就写得很快。小菜一碟。

## 锯齿形遍历其实是在练普通层序

这道题表面上多了“锯齿形”，但我感觉它真正检查的还是普通层序遍历稳不稳。

如果层序遍历本身写得不熟，就会在队列什么时候 `pop`、孩子什么时候入队、当前层和下一层怎么分开这些地方卡住。现在这些步骤已经比较自然了，所以后面的 `reverse` 反而只是一个很薄的后处理。

我今天还顺手写了一个 DFS 版本。递归时带上 `level`，第一次到达某一层时先给 `ans` 开一个空数组，之后把节点值放到对应层里。最后同样翻转奇数层。这个写法和 BFS 不一样，但和昨天的 DFS 版层序遍历能接上：递归负责走完整棵树，`level` 负责把节点送回正确的层。

## 判断子结构，绕在两个递归函数里

第二道题是判断树的子结构，用递归实现。

这题整体思路不算难：外层递归负责在 `A` 这棵树里找可能的起点，内层递归负责从某个起点开始，判断 `B` 的结构和值能不能和 `A` 对上。也就是说，我不是只写一个递归函数就结束，而是把问题拆成了两层：

```cpp
bool isSubStructure(TreeNode* A, TreeNode* B);
bool match(TreeNode* A, TreeNode* B);
```

`isSubStructure` 更像是在问：“`B` 会不会从 `A` 的某个节点开始匹配？”  
`match` 问的是：“如果现在已经决定从这两个节点开始比，后面能不能一直比下去？”

这两个问题听起来接近，但边界条件完全不一样。最容易绕的地方就在空指针。

对外层判断来说，题目明确说空树不是任意一棵树的子结构，所以：

```cpp
if(B == nullptr) return false;
```

但对内层匹配来说，`B` 如果已经走空了，说明 `B` 的所有节点都匹配完了，这反而应该成功：

```cpp
if(B == nullptr) return true;
```

同样是 `B == nullptr`，一个返回 `false`，一个返回 `true`。这个对比挺容易把人绕晕，因为它不是在判断同一个问题。外层是在判断“空树算不算子结构”，内层是在判断“模式树是不是已经匹配完”。

## 边界条件不能只靠感觉

今天写这题时，我最大的感觉是：递归的主体往往不难，难的是出口到底在回答哪个问题。

如果只是盯着代码里的 `A` 和 `B` 看，很容易变成机械地试 `true` 和 `false`。我现在的经验有点朴素：先把这个递归函数正在回答的问题翻译成一句人话，再去写边界条件；如果还是拿不准，就用几个极端例子反复试一下，看看 `true` 和 `false` 哪个才符合题意。

这种“试”不是乱猜，更像是用反例把语义逼出来。比如空树、单节点树、`B` 已经匹配完但 `A` 还有节点、`A` 提前空了但 `B` 还有节点，这几个情况一摆出来，两个递归函数的出口就清楚多了。

今天两道题，一个让我确认层序遍历已经比前几天熟很多，另一个提醒我递归不能只看形状相似。下一步如果继续写树题，我想多留意每个递归函数的“问题定义”。只要这个定义没说清楚，边界条件就很容易变成靠手感撞答案。
