## 不断合并权值最小的节点

哈夫曼树的建立过程很清楚：每次从当前节点中找出频率最小的两个，把它们合并成一个新节点，再让新节点回到待处理的节点中。一直重复，最后剩下的节点就是根节点。

每轮用 `find_min_node` 找最小值，再通过交换把选中的节点放到数组末尾。两个节点合并以后，新节点放回仍然参与后续查找的位置。

```cpp
for(int i=1;i<n;i++){
    int index1 = find_min_node(node_arr,n-i);
    swap_node(node_arr,index1,n-i);
    int index2 = find_min_node(node_arr,n-i-1);
    swap_node(node_arr,index2,n-1-i);
    int new_freq = node_arr[n-i]->freq + node_arr[n-i-1]->freq;
    node* new_node = get_new_node(new_freq,0);
    new_node->lchild = node_arr[n-i-1];
    new_node->rchild = node_arr[n-i];
    node_arr[n-1-i] = new_node;
}
```

建好树以后，再递归遍历左右子树：向左走就在缓冲区追加 `0`，向右走就追加 `1`，遇到叶子节点时输出当前字符和路径。前几天练过二叉树遍历，这一部分写起来很自然。

## 我还是会被循环边界绕进去

今天真正让我想了很久的，是建立哈夫曼树时最外层的 `for` 循环。

到底从 `i` 还是 `i - 1` 开始算？条件应该写 `<` 还是 `<=`？这些看起来只是差一个数字，但一旦放进数组下标里，后面的表达式都会跟着变化。我以前写循环时就经常在边界上犹豫，今天又撞上了这个问题。

后来我还是回到过程本身来想：最开始有 `n` 个节点，每合并一次，待处理的节点总数减少 `1`。最后只需要留下 `1` 个根节点，所以总共要合并 `n - 1` 次。这样一来，`for(int i = 1; i < n; i++)` 就比较容易确定了，它刚好执行 `n - 1` 轮。

我现在觉得，写这类循环时不能只盯着下标试来试去。先问自己三个问题会更靠谱：

- 开始时有多少个元素？
- 每轮会减少或增加多少？
- 循环结束时应该剩下多少？

等这三个问题有了答案，再决定 `<` 还是 `<=`。我对这种循环边界还是不够熟练，但至少今天找到了一种更踏实的判断方法。
