# 2026-06-06：仓库选址，思路想明白以后还要落到代码里

今天的代码量算很少，只写了一道仓库选址的问题。题目本身不难：在一条数轴上有若干家商店，要选一个货仓位置，让每天从货仓到所有商店的距离总和最小。只要数学直觉稍微敏感一点，应该很快就能想到中位数。

真正的最简做法也确实是这样。先把所有坐标排序，选中位数作为货仓位置，再把每个点到中位数的距离加起来。因为在数轴上，货仓往左或往右偏离中位数时，总会让一边的点少受益、另一边的点多付出；到中位数时，两边的拉扯刚好最稳。

我理清这个思路以后，写代码时却没有直接按这个最简单的版本来。我用了左右指针，从中间往两边扩，分奇数和偶数两种情况，计算左右两端配对后的距离差。这个思路也能做出来，本质上是在利用排序后两两配对的距离：

```cpp
ans += abs(arr[left] - arr[right]);
```

比如样例 `1 3 5 6 10`，选中位数 `5` 时距离和是 `4 + 2 + 0 + 1 + 5 = 12`；如果从中间往两边配对，也可以算成 `(6 - 3) + (10 - 1) = 12`。两种算法算的是同一件事，只是后者代码绕了一点。

## 复杂的不是题目，而是我把它写复杂了

今天比较明显的问题不是我完全没有思路，而是我有了思路以后，没有马上问自己：“有没有更直接的写法？”

如果按中位数写，代码会非常短：排序，取 `arr[N / 2]`，遍历累加距离。奇数、偶数都不用单独分开讨论，因为偶数个点时，中间两个点之间的任何位置都能取得最小值，直接取右中位数或左中位数都可以。

我一开始用了左右指针，于是后面就不得不处理更多细节：奇数时两个指针从同一个中位数出发，偶数时两个指针从中间两个位置出发；每一轮先算还是先移动；移动方向到底是向外扩还是向里收。题目本身只是在考察中位数，但代码里被我引入了额外的状态。

这个过程让我意识到，算法题里的“想明白”有两层。第一层是知道数学性质或者核心结论，第二层是把它翻译成尽量稳定的代码。今天我第一层大概过了，第二层还不够稳。

## 从思路到代码，还得继续磨

我后来把几个问题一点点改掉：先补上排序，再把单个商店时的输出处理好，之后把结果从覆盖改成累加，最后修正偶数情况下指针扩展的方向。`ans` 也改成了 `long long`，避免距离和比较大时溢出。

这些错误单独看都不算难，但连在一起就说明一件事：当实现方式比必要的复杂时，我需要维护的细节会明显变多。不是说左右指针不能写，而是如果有一个更直接、更不容易错的中位数版本，我应该优先选它。代码越贴近问题本身，后面出错的空间就越小。

今天这道题算是一个提醒。以后遇到这种带有明显数学性质的题，我想先把最朴素、最短的实现写出来，再考虑有没有别的写法。能做出来当然重要，但更重要的是把“我知道怎么做”变成“我能稳定地写出来”。
