# 2026-06-06：从仓库选址到合并果子，二叉树这一章收尾

今天的代码量不算多，就做了两道题，不算难，一道是仓库选址，一道是合并果子。它们都不是传统意义上的二叉树遍历题，不过背后的想法和这几天学到的内容能接上：一个靠中位数，一个靠哈夫曼式的合并过程。

仓库选址那道题本身不难。把所有商店坐标排好序，选中位数作为货仓位置，再累加每个点到中位数的距离，就能得到最小距离和。这个结论用数学直觉其实很容易想出来：货仓往一边偏时，总会让另一边更多点付出额外距离，到中位数附近时两边的拉扯最平衡。

我真正卡住的地方还是实现。最直接的写法明明是取一个中位数再遍历求距离，我却写成了左右指针，从中间往两边扩，还要分奇数和偶数讨论。这个方法也能做，但代码状态变多了：指针从哪里开始、先算还是先移动、偶数时往哪个方向扩，都要额外照顾。最后虽然改对了，但这道题提醒我，想明白算法不等于已经找到了最稳的实现。

## 合并果子像是在模拟哈夫曼编码

后面又做了合并果子。它比仓库选址要难一些，但如果刚学过哈夫曼编码，思路反而很自然：每次选当前数量最小的两堆合并，把合并后的新堆放回去，直到只剩下一堆。每一次合并的代价都会继续影响后面的总代价，所以小的堆应该尽早合并，让它们在后续结构里承担更深的路径。

这和哈夫曼编码的感觉很像。频率小的字符可以放得深一些，频率大的字符应该尽量靠近根；合并果子里也是一样，每次把最小的两个重量合成一个新重量，本质上就是在构造一棵代价最小的哈夫曼树。

我自己的第一版实现比较传统：用 `vector` 存所有堆的重量，每一轮都重新排序，然后用下标拿到最小的两个元素。这个写法很直观，写起来也顺手：

```cpp
sort(weight.rbegin(), weight.rend());
int sum = weight[weight.size() - 1] + weight[weight.size() - 2];
```

但问题也明显。每合并一次就排序一次，而排序本身复杂度不低。`n` 稍微大一点时，这个版本虽然能模拟出正确过程，但效率不够好。这里我能感觉到自己还是习惯先写“看得见”的容器操作，再慢慢意识到数据结构本身可以帮我省掉很多重复工作。

## `set<pair>` 解决了重复元素的问题

听老师讲以后，我看到另一种写法：用 `set` 存当前所有堆。`set` 会自动按从小到大维护元素，最小值直接从 `begin()` 取出来，不需要每轮手动排序。

不过 `set` 有互异性，单纯把果子数量放进去会出问题。比如有两堆都是 `5`，如果只插入数字，集合里可能只保留一个 `5`。老师的处理方法是用 `pair<int, int>`，第一个值放果子数量，第二个值放编号：

```cpp
set<pair<int, int>> s;
s.insert(pair<int, int>(num, i));
```

这样即使两堆果子数量相同，编号不同，整个 `pair` 也不同，不会被过滤掉。这个细节挺有意思，因为它不是在改变算法，而是在顺着容器的性质补一个稳定的标识。算法想法和 STL 容器特性之间，其实也需要磨合。

AI 给出的版本用了优先队列。这个结构我还没有正式学到，属于下一章“堆与优先队列”的内容。它听起来会比 `set` 更贴近这道题，因为题目每次只关心当前最小的两个元素，不需要完整的有序集合。老师也说后面会用优先队列继续优化哈夫曼编码的实现，所以我今天只是先知道有这个方向，没有急着细看。

## 二叉树这一结束

到今天，二叉树章节的学习就正式结束了，当然后面肯定还会学习树的相关内容，明天就要开始学习“堆和优先队列”了。从最开始的节点定义、递归遍历、层序遍历，到线索化、序列化、哈夫曼编码，再到这些配套练习，很多东西一开始看着是零散的，后来慢慢能连起来。

今天这两道题也把一个问题摆得挺清楚：算法直觉、代码实现、容器选择是三件事。仓库选址里，我有了中位数直觉，但实现绕远了；合并果子里，我知道要每次取最小两堆，但一开始用的 `vector` 排序版本效率不够好。
