# 2026-05-29：二叉树遍历与线索化，第一次见识到树的威力

这两天基本都扑在二叉树的遍历与线索化上。

昨天主要是在看网课。老师讲前序、中序、后序遍历，讲线索二叉树里怎么废物利用，把空出来的指针域重新利用起来。由于期末复习时间紧张加上水课的事情也很多，一行代码都没写，只是过了一下理论。理论上好像能听懂：空的左指针可以指向前驱，空的右指针可以指向后继；为了区分它到底是孩子还是线索，还要加 `left_tag` 和 `right_tag`。但这些话停在 PPT 上的时候，还没有真正压到手上。

今天跟着老师把 `binary_tree_traversal&thread.cpp` 写了一遍，才发现这东西确实不简单。之前学链表、栈和队列，虽然也有指针和状态，但大多数时候还能顺着一条线想下去。树一出来，函数递归进左子树，再回到当前节点，再进右子树，脑子里就不能只盯着一个变量走了。每一层递归都有自己的 `root`，每一个 `root` 又代表一棵当前正在处理的子树。这个视角不切过来，代码就会看得很乱。

今天最先卡住的是新节点的初始化。`get_new_node` 里要把左右孩子都设成 `nullptr`，同时把 `left_tag` 和 `right_tag` 设成 `0`。一开始我会想：既然现在左右孩子都是空的，那为什么不等插入的时候再处理两边的tag？后来想清楚了，`tag = 0` 也不是说这条边已经有孩子，而是说这个指针域现在按“实际孩子边”来解释。等到线索化时，如果这一边仍然为空，才把它改成前驱或后继线索，再把对应的 tag 改成 `1`。

线索化本身更绕。中序线索化的代码必须把处理当前节点的逻辑放在两次递归中间：

```cpp
__build_inorder_thread(root->lchild);
// 处理当前节点
__build_inorder_thread(root->rchild);
```

这个位置看着像普通中序遍历里的打印位置，但它其实是整段线索化最关键的地方。因为我要建立的是“中序前驱”和“中序后继”，所以 `pre_node` 必须跟着中序访问顺序走。只有左子树处理完、当前节点第一次被访问到的时候，`pre_node` 才刚好是当前节点的中序前驱。把逻辑放到前面就变成前序关系，放到后面又变成后序关系，线索自然就错了。

我今天也第一次比较认真地体会到“二次封装”这个写法的用处。真正递归干活的是 `__build_inorder_thread`，外面的 `build_inorder_thread` 负责在开始前清空全局状态，在结束后处理最后一个节点的右线索。以前我可能会觉得这只是多包了一层函数，现在能看出它是在把“每次任务开始和结束时必须做的事”集中起来。比如 `pre_node` 是全局变量，程序启动时会初始化一次，但一次线索化结束后它已经指向中序序列的最后一个节点了。如果下次再线索化另一棵树，必须先把它重新置成 `nullptr`，不然新树的第一个节点会莫名其妙接到上一棵树的最后一个节点后面。

还有 `inorder_root` 这一点也挺妙：

```cpp
if(inorder_root == nullptr) inorder_root = root;
```

它不是专门去搜索最左节点，而是利用了中序遍历本身的访问顺序。第一次走到“访问当前节点”这一行时，当前 `root` 就是中序序列里的第一个节点。后面再访问到别的节点，`inorder_root` 已经不是空了，就不会再被改掉。这个地方让我感觉递归不是单纯地“调用自己”，它也能帮我自然地拿到某种顺序里的边界点。

今天虽然基本是全程跟着老师敲代码，但我没有只是把代码复制下来。tag 为什么一开始是 `0`，中序线索化为什么要插在两次递归中间，全局变量为什么每次任务前要重置，`inorder_root` 为什么能抓到第一个中序节点，这几个问题我都来回想了一遍。现在还谈不上完全熟练，但至少每个点背后在维护什么状态，我能说出个大概了。

这也算是我第一次见识到“树”的威力了，哈哈。它不像前面的结构那样只是在一个方向上移动，树里有子树、有递归返回、有访问顺序，还有线索化这种对指针域含义的重新解释。对数据结构的抽象理解、对递归的使用、对函数再封装的编程技巧，全都挤在这一小段代码里，够我消化一大段时间。
