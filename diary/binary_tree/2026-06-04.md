# 2026-06-04：重写层序遍历，才发现细节真的会咬人

今天主要是把昨天写过的几道二叉树题重新敲了一遍：二叉树层序遍历、从前序和中序还原二叉树、翻转二叉树、N 叉树前序遍历。除此之外，又补了一道自底向上的层序遍历。

自底向上的层序遍历有一说一不难。先按普通层序遍历把结果存成从上到下的顺序，最后用一次 `reverse` 翻转，就能得到题目要求的从叶子层到根节点的结果。它本身没有绕很远，反而像是把昨天的层序遍历又拿出来确认了一遍：如果普通层序能写稳，这题就只是多了一步后处理。

今天真正让我有感觉的，还是“重写”这件事。

## 跟着写时顺理成章，自己写时处处有坑

昨天跟着老师写的时候，很多地方我都觉得很自然。比如 BFS 层序遍历里先写：

```cpp
int cnt = q.size();
```

然后再让 `for` 循环跑 `cnt` 次。刚开始我甚至觉得这一步有点多此一举，直接把 `q.size()` 写进 `for` 条件里不也一样吗？

今天自己敲才发现，完全不一样。

队列不是静止的。遍历当前层的时候，我会把当前节点的左右孩子继续压进队列，所以 `q.size()` 在循环过程中一直变。如果直接用它控制循环，当前层和下一层的边界就会混在一起。`cnt` 的意义不是“少写几个字”，而是在进入这一层之前，把这一层有多少个节点先定死。只有这样，队列里新加入的孩子才会留到下一轮，也就是下一层再处理。

这个细节以前看答案时很容易滑过去。自己写的时候忘一次，才知道它其实是层序遍历的边界。

## `q.front()` 也不是随便放在哪里都行

另一个让我卡住的点，是 `q.front()` 应该放在哪里。

一开始我把取队头节点的语句想得太随意，好像只要最后能拿到节点就行。但 BFS 的循环每次处理的是队列当前的队头：访问它，把它的孩子压入队列，然后把它弹出。这个顺序一乱，后面的节点就可能没有按层完整进入队列。

现在我对这几步的顺序更清楚了：

```cpp
TreeNode* node = q.front();
q.pop();
```

拿到当前节点以后，再围绕这个节点处理值和左右孩子。`front` 和 `pop` 不是普通的取值语句，它们共同推进了队列。BFS 写起来像模板，但模板背后其实是在维护“当前层正在被消费，下一层正在被准备”这个过程。

## DFS 写层序遍历，关键是先开出这一层

今天还重新写了 DFS 版本的层序遍历。这个写法很巧：递归还是往深处走，但每次把当前层数 `level` 带下去，最后用 `ans[level]` 把节点放回对应层。

我和正确写法之间，差的就是这一句：

```cpp
if(level == ans.size()) ans.push_back(vector<int>());
```

之前看老师写的时候，我会觉得它很合理，但没有真的往心里去。今天自己写到 `ans[level].push_back(root->val)` 时才意识到，如果这一层的数组还没创建，直接访问 `ans[level]` 就会越界。

这句代码做的事情很朴素：第一次来到某一层，就先给这一层开一个空数组。后面再遇到同一层的节点，就直接追加进去。

它让我重新理解了 DFS 版层序遍历的核心。递归负责访问节点，`level` 负责告诉我节点属于哪一层，而 `ans.size()` 负责判断这一层有没有被创建。少了其中一步，代码看起来差不多，实际就站不住。

## 重写不是重复劳动

今天这几道题都不算难，甚至自底向上的层序遍历可以说很直接。但我反而更能体会到重写的意义。

跟着老师写的时候，我经常会默认每一句都是应该存在的。可一旦把代码遮住，自己从空白处开始写，那些“理所当然”的地方就会突然露出问题：为什么要固定 `cnt`，为什么队头节点要在循环里取，为什么 DFS 写层序之前必须先判断 `level == ans.size()`。

这些不是很大的算法创新，但它们决定了代码到底是在碰巧像答案，还是我真的理解了这个过程。

今天的收获不在于多刷了一道题，而在于我重新看见了层序遍历里那些容易被忽略的边界。下一步还是继续重写。尤其是 BFS 和 DFS 两种写法，我想再多敲几遍，让它们不只是“看懂”，而是能自然地从手里写出来。
